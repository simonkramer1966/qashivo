### 🟡 Issue 3: Index Creation Missing Size Estimates

**Good:** Proper indexes identified based on query patterns
**Missing:** Expected index size (important for database capacity planning)

**Recommendation:**

```sql
-- Before creating indexes, estimate their size:
SELECT 
  schemaname,
  tablename,
  pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size
FROM pg_tables 
WHERE tablename IN ('invoices', 'actions', 'contacts', 'emailMessages', 'outcomes', 'attentionItems')
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC;

-- Rule of thumb: indexes typically add 20-30% to table size
-- If invoices table is 100MB, expect ~20-30MB for index on (tenant_id, status)
```

**Expected total index overhead:** ~50-100MB (acceptable)

**Add this to Phase 4 verification steps.**

## What Could Still Go Wrong (& How to Handle It)

### Scenario 1: A "safe" table deletion breaks something
**Symptoms:** After Phase 1B, a feature stops working
**Cause:** Reference was missed in the audit (edge case: dynamic table name, commented code that's actually used)
**Response:**
1. Check git diff to identify which table was deleted
2. Restore from checkpoint immediately
3. Search codebase more thoroughly for that table name
4. Move table to "keep" list
5. Document why the reference was missed

### Scenario 2: Service consolidation introduces subtle bug
**Symptoms:** Decision engine produces different recommendations
**Cause:** Hidden state dependency or timing assumption
**Response:**
1. Rollback Phase 2B commit
2. Compare before/after code side-by-side
3. Identify state initialization or timing difference
4. Fix in isolation with unit tests
5. Re-attempt consolidation

### Scenario 3: N+1 fix makes queries slower
**Symptoms:** Batch query takes longer than individual queries
**Cause:** Database query planner chooses bad execution plan for IN clause
**Response:**
1. Run `EXPLAIN ANALYZE` on the slow query
2. Check if indexes are being used
3. Consider adding covering index
4. If still slow, revert to original approach
5. Document in code why optimization was reverted

### Scenario 4: Index creation takes >10 minutes
**Symptoms:** `CREATE INDEX` command hangs
**Cause:** Lock contention or data corruption
**Response:**
1. Check for long-running transactions: `SELECT * FROM pg_stat_activity WHERE state='active';`
2. Kill blocking queries if safe
3. If data corruption suspected, skip index creation and investigate
4. Document issue for DBA review
