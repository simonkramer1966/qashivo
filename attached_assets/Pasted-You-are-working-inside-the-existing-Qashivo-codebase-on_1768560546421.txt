C) OUTBOUND SENDING PIPELINE

1) Create/extend a service: sendgridEmailService.ts
Function: sendActionEmail(actionId: string): Promise<{emailMessageId}>
Steps:
- Load Action + related Customer + Invoice (if any) + SME/Partner.
- Assert Action.status is APPROVED (or equivalent).
- Build tracking + reply routing token:
  - token = base64url( actionId + "." + customerId + "." + (invoiceId||"none") + "." + randomShort )
  - store token on Action (new field action.emailReplyToken) OR store on EmailMessage.
- Set Reply-To to: `${EMAIL_REPLY_PREFIX}+${token}@${EMAIL_REPLY_DOMAIN}`
- Also include custom headers:
  - X-Qashivo-Action-Id: <actionId>
  - X-Qashivo-Customer-Id: <customerId>
  - X-Qashivo-Invoice-Id: <invoiceId or empty>
  - X-Qashivo-Thread-Key: <threadKey>
- Determine threadKey:
  - If invoiceId exists: `inv_<invoiceId>`
  - Else: `cust_<customerId>`
- Subject rules:
  - If invoice: `Invoice <invoiceNumber> — <CompanyName>`
  - Else: `Quick reminder — <CompanyName>`
- Body rules:
  - Use existing template system if present; otherwise implement a minimal templater:
    variables: {{customerName}}, {{companyName}}, {{invoiceNumber}}, {{amountDue}}, {{dueDate}}, {{payLink}}
- Create EmailMessage row with direction OUTBOUND, status QUEUED.
- Send via SendGrid:
  - Use @sendgrid/mail if already installed; otherwise add it.
  - Capture sendgrid message id if available (some libs provide it in response headers).
- Update EmailMessage status SENT, sentAt now.
- Update Action.executedAt now; Action.status EXECUTED (or existing).
- Append a Timeline event if Qashivo has an activity/timeline model already.

2) Hook into existing execution flow:
Wherever Qashivo “executes” approved actions, ensure EMAIL actions call sendActionEmail(actionId).
If no executor exists yet, create:
POST /api/actions/:id/execute
- Auth required, role allowed.
- Calls sendActionEmail and returns success.

D) INBOUND: SENDGRID INBOUND PARSE WEBHOOK
========================================================
Implement endpoint:
POST /api/webhooks/sendgrid/inbound
This must accept multipart/form-data SendGrid payload fields such as:
- from, to, subject, text, html, headers
- attachments info (optional)
- envelope (json) sometimes
- Inbound Parse may include "attachments" and multiple attachment files.

Steps:
1) Parse multipart safely:
- Use existing middleware (busboy/multer/nextjs body parser off).
- Store attachments meta only for MVP (filename, contentType, size). Do not store raw files unless easy; if stored, put in object storage later.

2) Identify the token and link records:
Primary method (preferred):
- Extract recipient address from "to"
- Find pattern: `${EMAIL_REPLY_PREFIX}+<token>@${EMAIL_REPLY_DOMAIN}`
- If token found:
  - decode token to get actionId/customerId/invoiceId (if you encoded them)
  - else: look up EmailMessage where replyToken==token or Action.emailReplyToken==token
Fallback methods:
- Read headers for X-Qashivo-Action-Id etc.
- Parse In-Reply-To / References and match to EmailMessage.sendgridMessageId (best-effort).

3) Create inbound EmailMessage:
- direction INBOUND, status RECEIVED initially
- set customerId/invoiceId/actionId based on match
- store inbound fields (from/to/subject/text/html/headers json)
- set receivedAt now

4) Outcome detection (MVP heuristic + placeholder for LLM):
Create a function detectOutcomeFromText(text: string, subject: string): { outcomeType, promiseDate?, confidence, notes, needsReview }
MVP heuristic rules (fast + deterministic):
- If contains “paid” / “payment made” / “sent the transfer” => ALREADY_PAID (confidence 0.7)
- If contains “dispute” / “wrong” / “not ours” / “query” / “credit note” => DISPUTE (0.7)
- If contains a date pattern (e.g. 12/02/2026, 12 Feb) plus phrases “will pay” / “can pay” => PROMISE_TO_PAY with parsed date (0.75)
- If contains “more time” / “next week” / “end of month” => REQUEST_MORE_TIME (0.6)
- If contains “payment plan” / “installments” => PAYMENT_PLAN_REQUEST (0.7)
- If contains “not the right person” / “no longer here” => WRONG_CONTACT (0.8)
- Else UNKNOWN (0.3)

Date parsing:
- Implement robust parsing for UK formats (DD/MM/YYYY, “12 Feb”, “12th February”).
- If relative dates like “next Friday”, convert using Europe/London timezone.

5) Create DetectedOutcome linked to inbound EmailMessage.
- Set needsReview=true if confidence < 0.65 or outcomeType==UNKNOWN
- status on EmailMessage becomes PARSED

6) Update business state (minimal, safe):
- If PROMISE_TO_PAY with date:
  - Update Invoice.expectedPaymentDate and Invoice.intentConfidence (if fields exist)
  - OR create a separate “InvoiceIntent” record if your system already models intent.
- If DISPUTE or PAYMENT_PLAN_REQUEST:
  - Flag Invoice as “needs attention” / create Exception record if that exists.
Do NOT auto-close invoices.

7) Respond 200 quickly:
Return { ok: true } within <2 seconds. Do not do heavy processing inline.
If the codebase already uses a job queue, enqueue parse. If not, keep it inline but lightweight.
