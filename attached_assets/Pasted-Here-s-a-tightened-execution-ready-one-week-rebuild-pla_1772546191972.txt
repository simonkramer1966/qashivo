Here’s a tightened, execution-ready one-week rebuild plan that matches your objectives, with daily goals (Mar 3 → Mar 10 demo), acceptance checks, and a few gotchas to prevent demo-day surprises.

Calendar plan (Europe/London)
Day 1 — Tue Mar 3: Server stability (T001)

Goal: server never “mysteriously dies”, and restarts never get stuck on a port.

Do:

Add server.on('error', ...) handling for EADDRINUSE with loud logging + process.exit(1)

Add graceful shutdown on SIGTERM + SIGINT that server.close() then exits

Add uncaughtException + unhandledRejection guards (log + shutdown)

Acceptance checks (do these locally + on deployed env):

Restart loop: start → stop → start 10 times, no EADDRINUSE

Kill test: kill -TERM <pid> and Ctrl+C both log clean shutdown and exit 0/1 predictably

Forced error: throw from a request handler → see log + shutdown (no silent hang)

Demo-risk reducer: ensure you log which PID and which port at startup and shutdown.

Days 2–4 — Wed Mar 4 to Fri Mar 6: Backend refactor (T002–T013)

Goal: split routes without changing any behavior.

Golden rule: no API changes, no path changes, no auth changes, no response shape changes.

Practical extraction order (minimize merges/conflicts)

T002 contacts

T003 invoices

T006 integrations (Xero)

T005 communications (email/SMS/voice/retell)

T004 collections/actions (biggest)

T007 settings/RBAC

T008 dashboard/analytics

T009 banking/cashflow

T010 outcomes/payment plans

T011 onboarding/demo-data

T012 inbox/inbound

T013 remainder cleanup + diff audit

Acceptance checks after each extraction

“Register count” check: number of routes in the module matches what you moved

Smoke calls: hit a small set of endpoints per module (200 + correct JSON shape)

Build + typecheck + server boot

Diff audit at the end: every old route appears exactly once across new modules

Tip: When you move routes, keep them as a single exported function:

export function registerContactRoutes(app: Express) { ... }
so server/index.ts becomes a clean “route registrar”.
