--- Qashivo Investor MVP — Source of Truth (UPDATED) ---

A. Required capabilities (MVP-01..MVP-26)

MVP-01 Promise-to-Pay (PTP) lifecycle
- Capture PTP on an invoice; statuses: active | kept | broken | replaced | cancelled.
- Auto-mark kept/broken from Payments.
- Store promised_date, created_ts, source_message_id, created_by, kept_ts.

MVP-02 Promise Reliability Score (PRS)
- Exponential decay half-life = 60 days across past PTP outcomes.
- PRS_mean = (1 + successes_d) / (2 + successes_d + failures_d).
- Compute Wilson lower bound @95% (PRS_LB95) for conservative routing.

MVP-03 Guardrails (baseline)
- Suppress dunning when dispute_active OR when PTP is active (before promised_date).

MVP-04 Escalation rule
- If PRS_LB95 < 0.40 OR broken_ptp_in_30d >= 1 → create “senior_call in 24h”; suppress templated nudges.

MVP-05 Predicted pay date v0
- P50 = due_date + expected_slip_days(buyer).
- P90 = P50 + segment_q90_slip.
- If active PTP exists, clamp P50 to promised_date unless history shows consistent slip.

MVP-06 90‑day cash-in curve
- Portfolio roll-up using per-invoice P50/P90 to plot P50 band and P90 band over next 90 days.

MVP-07 Explainability
- Every recommended action shows 2–3 reasons (e.g., “PRS_LB95=0.31; 1 broken PTP in 30d; sentiment down”).

MVP-08 Prioritized Worklist
- Sort by: high value & low PRS, broken PTP, approaching PTP date, then others.

MVP-09 Invoice Detail screen
- Invoice status, dispute badge, PTP panel, predicted pay date, next action, and “why”.

MVP-10 Portfolio Dashboard
- Cash curve (P50/P90), tiles: PTP kept rate, average slip days, mis-fires count (dunning during dispute/after payment).

MVP-11 “Insured” flag & buyer limit (stub)
- Toggle at invoice level; when ON, show indemnity % (e.g., 90%) and buyer limit.

MVP-12 Spot finance offer (stub)
- If invoice undisputed & insured flag ON, show a partner-backed offer card: advance %, fee, tenor note. No real underwriting.

MVP-13 Recovery timeline (stub)
- For EN/Wales: show earliest dates for PAP Letter (sole trader) or LBA (company), MCOL issue date, and post-judgment transfer-up eligibility (>£600). Buttons disabled with tooltip “available post-funding”.

MVP-14 Messaging guardrails (basic)
- Send window (08:30–17:30 local, weekdays only), max 3 touches/week, respect opt-out flag.

MVP-15 Templates & call script (minimal)
- Two email templates: gentle_nudge, firm_ptp_plus_consequence.
- One **voice call script** suitable for AI agent.

MVP-16 Demo dataset & toggles
- Seed data (~10 customers / 120 invoices) covering predictable-late vs serial-promiser behaviors.
- UI toggles to: set dispute on/off, add PTP, mark broken PTP, flip insured flag.

--- NEW multi-channel + webhook capabilities ---

MVP-17 Webhook ingestion (SMS + email)
- HTTP endpoints to receive provider webhooks containing: message_id, thread_id, customer_id, invoice_id?, ts, channel, direction, actor, text, intents[], sentiment, provider_signature.
- Store as Comm events; update response_latency, sentiment trend, and allow these events to trigger playbook rules.

MVP-18 Outbound AI voice calls (integration)
- Ability to schedule an AI voice call action from the playbook.
- Receive **voice webhooks** for call events: started, connected, voicemail, completed, failed, transcript_url, summary, detected_intents[], detected_sentiment, ptp_captured (date?), call_result (spoke/voicemail/failed).
- If ptp_captured=true, create/attach a PTP with source="voice" and promised_date parsed from payload or transcript.

MVP-19 Channel policy & throttling v2
- Per-channel budgets and windows (Email, SMS, Voice): max attempts/week and time-of-day windows.
- Fallback order (e.g., Email → SMS → Voice) configurable; prevent voice if customer opted out or has recent “do-not-call” flag.

MVP-20 Webhook security & idempotency
- Verify provider signatures (e.g., HMAC); reject on fail.
- Idempotency keys to avoid duplicate actions on retry.

MVP-21 Explainability (channel-aware)
- Next-action reasons must reference channel logic (e.g., “Voice chosen: PRS_LB95 low, broken PTP, email unresponsive 10d, within call window”).
- Customer 360 shows latest **voice transcript snippet/summary** and sentiment.

MVP-22 Outcome logging for voice
- Record talk path: attempts, connect status, duration_s, result, who spoke to, and whether a PTP or dispute was captured.
- Use outcomes to update PRS/RHS.

MVP-23 Opt-out/blacklist per channel
- Fields and enforcement for email_opt_out, sms_opt_out, voice_opt_out/do_not_call; UI badges and hard guardrails.

MVP-24 Inbound reply threading
- Map inbound SMS/email/voice callbacks to existing threads by thread_id or heuristic; update response_latency metrics.

MVP-25 Test harness for webhooks (dev)
- CLI or test endpoint to replay canned webhook payloads for SMS/email/voice so demos are deterministic.

MVP-26 Metrics for multi-channel
- Tiles: call connect rate, PTP captured via voice %, % actions blocked by guardrails, webhook error rate.

B. Minimal architecture assumptions (adapt mapping if stack differs)

D. Acceptance tests this MVP must pass live (add to AC-1..AC-6)

AC-2: Log a PTP for Friday → cadence shifts; if unpaid by Monday, a broken‑promise task appears.
AC-3: Two customers send the same “We’ll pay Friday” → one escalates (low PRS), one de‑prioritizes (predictable-late).
AC-4: Cash curve updates when PRS/PTP status changes.
AC-5: Finance card changes when Insured toggle flips.
AC-6: Send window and max touches correctly throttle outbound.
AC-7: **Voice webhook** with ptp_captured=true creates a PTP and suppresses further dunning; Explainability shows “captured via voice”.
AC-8: **Inbound SMS/email webhook** updates sentiment and response_latency; RHS alert triggers AM outreach when slope falls.
AC-9: **Channel budgets** enforce max voice attempts/week and voice time window; attempts beyond limits are blocked and logged.
AC-10: **Webhook idempotency**: replaying the same provider payload (same idempotency_key) does NOT duplicate PTPs or actions.
