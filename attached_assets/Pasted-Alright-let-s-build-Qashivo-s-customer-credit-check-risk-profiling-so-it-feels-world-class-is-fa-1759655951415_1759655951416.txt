Alright, let’s build Qashivo’s customer credit-check + risk-profiling so it feels world-class, is fast on mobile, and pushes cleanly into Xero.

1) Objectives (what “great” looks like)

One smooth flow to add a customer, assess risk, set a credit limit and payment terms, then sync to Xero.

Modular scoring (data + heuristics + machine learning later) with explainable decisions.

Auditable (who approved what, when, why) with quick re-assess and version history.

Friction-aware UX: defaults + auto-fill + progressive disclosure so it’s 2–4 minutes for a typical SME.

3) Risk Scoring Model (starter, transparent & tunable)

Score = sum(weights × features), capped 0–100

Feature	Rule	Points (example)
Company age	≥3y +10; 1–3y +4; <1y 0	
Filings on time	Yes +8; No -6	
Legal/adverse (CCJs)	None +8; Minor (<=1, paid) 0; Active -20	
Bureau score (0–100)	Normalize to 0–30 points	
Director churn (12m)	0–1 changes +4; ≥2 changes -6	
Sector risk (benchmark PD)	Low +6; Medium 0; High -8	
Trade pay. behaviour (DBT)	Early/On time +10; 1–10 DBT +4; >10 DBT -8	
Internal payment history	None 0; 1–2 late -4; >2 late -10	

Bands

A (80–100): Very low risk → generous limit, NET30 ok

B (65–79): Low risk → moderate limit, NET14–30

C (50–64): Medium → modest limit, NET7–14 or upfront % deposit

D (35–49): High → low limit, Due on Receipt or DD required

E (<35): Very high → prepaid only or decline

Credit Limit formula (starter)

BaseLimit = min( 1.5 × AvgMonthlySales , PolicyCap )
Multiplier by band:  A 1.2×, B 1.0×, C 0.6×, D 0.3×, E 0×
FinalLimit = round_to_nearest_100( BaseLimit × Multiplier )
Hard clamps: if adverse active → max(£1,000 or 10% of BaseLimit)


Payment Terms by band

A: NET30 (or org default)

B: NET14–30

C: NET7–14 (+ consider 20% deposit)

D: Due on receipt or NET7 with DD mandate

E: Prepaid only

Conditions engine (examples)

If Band C and AvgInvoice > £10k → 25% deposit.

If Band D and DBT > 15 → DD required + limit ≤ £2k.

If any legal adverse open → manual review.

6) Xero sync details (safe, flexible approach)

Create/Update Contact with: name(s), email(s), phones, addresses, tax number, website, default currency.

Payment terms:

If your Xero setup supports contact-level defaults, set them via the Contacts API.

If not, store the terms in Qashivo and whenever you create a Xero invoice via API, apply those terms on the invoice so the due date is correct.

Risk tags:

Use Contact Groups like Risk_A, Risk_B, etc. If groups aren’t used in your org, prefix in the Account Number field (e.g., QA-RiskA-CL24000) or add a small PDF summary as an attachment.

Idempotency: store Xero ContactID after first creation to avoid duplicates.

Merges: if Xero has a near-match, show the merge dialog above rather than creating a new contact.
